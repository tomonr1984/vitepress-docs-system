# ドキュメンテーション標準

このドキュメントでは、ツールドキュメンテーション管理リポジトリにおけるドキュメント作成の標準とガイドラインを定義します。

## 目的

- 一貫性のあるドキュメント品質を保証
- 読みやすく使いやすいドキュメンテーションを提供
- メンテナンスしやすい構造を維持
- 多言語対応の品質を保持

## ドキュメント構造の標準

### 必須セクション

すべてのツールドキュメントは以下の5つのセクションを含む必要があります：

1. **概要（Overview）** - `index.md`
2. **インストール（Installation）** - `installation.md`
3. **使用ガイド（Usage Guide）** - `usage.md`
4. **APIリファレンス（API Reference）** - `api.md`
5. **トラブルシューティング（Troubleshooting）** - `troubleshooting.md`

### ディレクトリ構造

```
docs/
├── tools/
│   └── [tool-name]/
│       ├── index.md
│       ├── installation.md
│       ├── usage.md
│       ├── api.md
│       └── troubleshooting.md
└── en/
    └── tools/
        └── [tool-name]/
            ├── index.md
            ├── installation.md
            ├── usage.md
            ├── api.md
            └── troubleshooting.md
```

## コンテンツ標準

### 1. 概要ページ（index.md）

**必須要素:**
- ツール名と簡潔な説明
- 主な機能の箇条書き（3-6項目）
- クイックリンクセクション
- 使用例（3-5項目）
- 行動喚起（CTA）

**構造テンプレート:**
```markdown
# [Tool Name]
[Brief description]

## 概要
[Detailed overview]

## 主な機能
- 🚀 **Feature 1**: Description
- 🔧 **Feature 2**: Description

## クイックリンク
- [Installation](installation.md)
- [Usage](usage.md)
- [API Reference](api.md)
- [Troubleshooting](troubleshooting.md)
```

### 2. インストールページ（installation.md）

**必須要素:**
- 前提条件
- 複数のインストール方法
- インストール確認手順
- 初期設定手順
- よくある問題と解決方法

### 3. 使用ガイドページ（usage.md）

**必須要素:**
- クイックスタート例
- 基本的な使用方法（2-3項目）
- よく使用されるワークフロー（2-3項目）
- 実用的な例（2-3項目）
- ベストプラクティス（3-5項目）

### 4. APIリファレンスページ（api.md）

**必須要素:**
- コマンド概要
- グローバルオプション表
- 各コマンドの詳細仕様
- 設定ファイル情報
- 環境変数一覧
- 終了コード一覧

### 5. トラブルシューティングページ（troubleshooting.md）

**必須要素:**
- よくある問題（3-5項目）
- プラットフォーム固有の問題
- エラーメッセージと対処法
- デバッグ方法
- サポート情報

## 文章スタイルガイド

### 基本原則

1. **明確性**: 曖昧な表現を避け、具体的で明確な言葉を使用
2. **簡潔性**: 冗長な表現を避け、必要な情報を効率的に伝達
3. **一貫性**: 用語、表現、構造の統一
4. **実用性**: 実際の使用シナリオに基づいた内容

### 日本語スタイル

- **敬語**: 「です・ます」調を使用
- **句読点**: 「、」「。」を適切に使用
- **カタカナ語**: 一般的なものは使用可、専門用語は初出時に説明
- **英単語**: 必要に応じて併記（例：「設定（configuration）」）

### 英語スタイル

- **時制**: 現在形を基本とする
- **人称**: 二人称（you）を使用
- **語調**: フレンドリーだが専門的
- **略語**: 初出時に定義（例：CLI (Command Line Interface)）

## コードとコマンドの表記

### コードブロック

```markdown
# 言語を必ず指定
```bash
command --option value
```

```javascript
const example = 'code';
```
```

### インラインコード

- コマンド: `command`
- ファイル名: `filename.ext`
- 設定項目: `config_option`
- 変数: `$VARIABLE`

### プレースホルダー

- 角括弧を使用: `[PLACEHOLDER]`
- 説明的な名前を付ける: `[YOUR_USERNAME]`

## 視覚的要素

### 見出し構造

```markdown
# H1: ページタイトル（各ページに1つのみ）
## H2: 主要セクション
### H3: サブセクション
#### H4: 詳細項目（必要に応じて）
```

### 箇条書きと番号付きリスト

**箇条書き（順序なし）:**
```markdown
- 項目1
- 項目2
  - サブ項目2.1
  - サブ項目2.2
```

**番号付きリスト（手順）:**
```markdown
1. ステップ1
2. ステップ2
3. ステップ3
```

### 表

```markdown
| 項目 | 説明 | デフォルト |
|------|------|-----------|
| option1 | 機能1の説明 | `default1` |
| option2 | 機能2の説明 | `default2` |
```

### 強調とハイライト

- **重要な情報**: `**太字**`
- *軽い強調*: `*イタリック*`
- `コード要素`: バッククォート
- > 引用やノート: 引用記法

### アイコンと絵文字

一貫した絵文字を使用：
- 🚀 機能・パフォーマンス
- 🔧 設定・カスタマイズ
- 📊 データ・分析
- 🎯 ターゲット・精度
- 📚 ドキュメント・学習
- 🌐 ネットワーク・多言語
- ⚠️ 警告・注意
- ✅ 成功・確認
- ❌ エラー・失敗

## リンクとナビゲーション

### 内部リンク

```markdown
- 相対パス: [使用ガイド](usage.md)
- 絶対パス: [概要](/tools/my-tool/)
- アンカーリンク: [セクション](#section-title)
```

### 外部リンク

```markdown
- [外部サイト](https://example.com)
- 新しいタブで開く場合は明記
```

### ナビゲーション構造

各ページの最後に関連ページへのリンクを追加：

```markdown
## 関連情報

- [前のページ](previous.md) - 前のページの説明
- [次のページ](next.md) - 次のページの説明
- [APIリファレンス](api.md) - 詳細なコマンドリファレンス
```

## 多言語対応標準

### コンテンツの対等性

- 日本語版と英語版で同等の情報を提供
- 文化的な文脈に配慮した内容調整
- 各言語での自然な表現を優先

### ファイル構造の一致

```
docs/tools/my-tool/index.md ←→ docs/en/tools/my-tool/index.md
```

### 言語間リンクの整合性

- パス構造の統一
- ナビゲーション要素の対応
- アンカーリンクの一致

## 品質チェックリスト

### コンテンツ品質

- [ ] すべての必須セクションが含まれている
- [ ] 情報が正確で最新である
- [ ] 実用的な例が含まれている
- [ ] 読みやすい構造になっている
- [ ] 一貫したスタイルが使用されている

### 技術的品質

- [ ] すべてのリンクが正しく動作する
- [ ] コードサンプルが動作する
- [ ] マークダウン記法が正しい
- [ ] 多言語版が同期している
- [ ] レスポンシブデザインに対応している

### アクセシビリティ

- [ ] 見出し構造が論理的である
- [ ] 代替テキストが適切に設定されている
- [ ] 色に依存しない情報伝達
- [ ] 適切なコントラスト比

## レビュープロセス

### 作成段階

1. テンプレートの使用
2. プレースホルダーの置換
3. コンテンツの作成
4. 自己レビュー

### レビュー段階

1. 技術的正確性の確認
2. スタイルガイドの遵守確認
3. リンクとナビゲーションの確認
4. 多言語版の同期確認

### 公開前チェック

1. ローカルビルドテスト
2. 全ページの動作確認
3. モバイル表示の確認
4. パフォーマンステスト

## メンテナンスガイドライン

### 定期更新

- 情報の正確性確認（四半期ごと）
- リンクの有効性確認（月次）
- ツールバージョンとの同期

### 品質向上

- ユーザーフィードバックの反映
- アナリティクスデータの活用
- ベストプラクティスの更新

### 一貫性維持

- 新機能追加時の全体調整
- スタイルガイドの遵守確認
- 多言語版の同期維持

この標準に従うことで、高品質で一貫したドキュメンテーションを維持し、ユーザーにとって価値のあるリソースを提供できます。
